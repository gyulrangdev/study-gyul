### 5주차 RAG 활용하기

#### 임베딩 모델

임베딩은 텍스트를 수치로 변환하는 작업을 의미함. 이 작업을 위해 사전 학습된 모델이 활용되며, 대표적인 모델로는 BERT가 있음.

RAG 작동 단계에서 임베딩은 두 차례 활용됨:

1. 문서 청크를 벡터 DB에 저장할 때
2. 사용자의 질문에 답할 근거를 벡터 DB에서 검색할 때
   - 사용자의 질문을 임베딩으로 변환 → 벡터 DB상에서 가장 유사도가 높은 청크를 추출

키워드 검색 대신 임베딩 검색을 사용하는 이유:

- 키워드 검색은 많은 제약을 가짐:
  - 사용자 질문에 적절한 키워드가 포함되어야 함
  - 사용자 질문이 N개의 키워드로 대표되어야 함
  - 문서 내에 해당 키워드를 정확히 포함하는 근거가 존재해야 함
- 임베딩 검색은 의미적 유사성을 파악할 수 있어 더 유연함

**임베딩 모델의 종류:**

- **BERT 계열**:

  - Google에서 2018년 개발한 기본 양방향 인코더 모델
  - 트랜스포머의 인코더만을 활용하여 구축
  - 위키피디아/BooksCorpus 등 레이블이 없는 텍스트 데이터로 사전 훈련된 언어 모델
  - 문장 임베딩에 특화된 Sentence-BERT(SBERT) 기반 라이브러리인 SentenceTransformers가 대표적
  - 단어들의 의미, 문장 내 단어의 역할, 문장의 맥락, 감정 등 다양한 정보를 학습하여 수치화

- **OpenAI 모델**:
  - text-embedding-3-small: 1536차원, 저비용 고효율 모델
  - text-embedding-3-large: 3072차원, 고성능 임베딩 모델

**오픈소스 vs 클로즈드소스 임베딩 모델:**

오픈소스 장점:

- 비용: 직접 호스팅할 경우 API 호출 비용 없음
- 커스터마이징: 자체 데이터로 파인튜닝 가능
- 투명성: 모델 아키텍처와 학습 방식 확인 가능
- 독립성: 외부 서비스에 의존하지 않음
- 개인정보 보호: 데이터가 외부 서비스로 전송되지 않음

오픈소스 단점:

- 인프라 요구사항: 모델 호스팅을 위한 컴퓨팅 자원 필요
- 유지보수: 직접 업데이트 및 관리 필요
- 기술적 난이도: 설정 및 최적화에 전문 지식 필요
- 성능: 일부 상용 모델보다 성능이 떨어질 수 있음

클로즈드소스 장점:

- 성능: 일반적으로 최고 수준의 성능 제공
- 확장성: API를 통해 쉽게 확장 가능
- 유지보수 불필요: 제공업체가 모델 업데이트 및 유지관리
- 편의성: 간단한 API 호출로 사용 가능
- 전문 지원: 기업용 서비스의 경우 기술 지원 제공

클로즈드소스 단점:

- 비용: API 호출에 따른 지속적인 비용 발생
- 의존성: 외부 서비스에 의존
- 개인정보 우려: 데이터가 외부 서비스로 전송됨
- 투명성 부족: 모델 구현 세부 사항 비공개

임베딩의 장점:

- 벡터 간 거리(유사도)를 측정함으로써 서로 다른 벡터 간 유사성을 파악할 수 있음
- 코사인 유사도를 통해 텍스트 임베딩 간의 거리를 측정함
- 임베딩 모델의 사전 학습 데이터셋에 따라 특정 언어의 문장 임베딩 성능이 결정됨

#### 벡터 DB

벡터 DB는 비정형 데이터를 저장하고 검색하는 데 특화된 데이터베이스임. 텍스트나 이미지와 같은 비정형 데이터는 데이터마다 갖고 있는 의미가 완전히 다르기 때문에 테이블 형태로 적재하기 어려워 벡터 DB를 사용함.

**벡터 DB vs RDB 차이점:**

1. 벡터 DB는 고차원의 벡터 데이터를 효율적으로 저장하고 검색하도록 설계됨
2. 데이터 조회 방식이 다름:
   - RDB: SQL 쿼리 기반으로 정확한 행과 열을 추출하여 정보 조회
   - 벡터 DB: 벡터 유사도 계산을 통해 데이터 조회, 정확히 일치하는 것보다 '비슷한 것'을 찾는 데 특화

**벡터 DB의 검색 방식:**

- 벡터 DB에 저장된 임베딩 벡터와 사용자 질문의 임베딩 벡터 간 유사도 계산을 통해 가장 높은 유사도를 가진 K개의 벡터를 찾음
- ANN(Approximate Nearest Neighbor) 알고리즘을 바탕으로 벡터 간 유사도를 계산
- 주요 ANN 알고리즘:
  - HNSW(Hierarchical Navigable Small World): 계층적 구조를 이용해 빠른 검색 가능
  - SPTAG: 공간 분할 트리와 그래프를 결합한 방식, 데이터를 트리 구조로 분류하고 그래프를 통해 세부 검색 수행

**벡터 DB의 종류:**

- **순수 벡터 DB**:

  - 설계 시점부터 임베딩 벡터 저장과 검색에 초점을 둠
  - 오픈소스: Chroma, Weaviate, Qdrant
  - 유료 클로즈소스: Pinecone, Ziliz
  - 장점: 벡터 DB로 갖춰야 할 기본적인 기능들이 가장 많이 발달됨
  - 단점:
    - 기존 DB와 결합이 어려움
    - CRUD 같은 기본 기능이 전통 DB에 비해 부실할 수 있음
    - 벡터 데이터 인덱싱 계산 과정이 무거워 새로운 데이터 추가와 활용이 느리거나 비용이 많이 들 수 있음

- **텍스트 전용 DB**:

  - Elasticsearch, Opensearch 등
  - 장점:
    - 성숙한 텍스트 검색 및 분석 기능과 벡터 검색을 결합
    - 복잡한 쿼리와 필터링 지원
    - 대규모 클러스터 배포에 대한 풍부한 경험과 도구
  - 단점:
    - 순수 벡터 DB에 비해 벡터 검색 성능이 다소 떨어질 수 있음
    - 더 많은 리소스가 필요할 수 있음

- **벡터 라이브러리**:

  - 벡터의 유사도 계산이나 클러스터링을 위한 라이브러리
  - FAISS, Annoy, Hnswlib
  - 단점:
    - DB로서의 기능이 부실
    - 메타데이터 필터링 기능 지원 없음
    - 시스템 유지보수가 원활하지 않을 수 있음

- **벡터 기능이 추가된 NoSQL**:

  - MongoDB Atlas: MongoDB의 클라우드 서비스에 벡터 검색 기능 추가
  - Redis: RedisSearch 모듈을 통해 벡터 검색 지원

- **벡터 저장 및 검색 가능한 SQL DB**:
  - PostgreSQL + pgvector: 오픈소스 확장 모듈로 벡터 검색 기능 추가
  - MySQL: 8.0 이상에서 공간 인덱싱을 활용한 제한적인 벡터 검색 지원
  - 장점:
    - 트랜잭션 지원, 강력한 쿼리 언어 등 RDBMS의 모든 장점을 유지
    - 기존 데이터와 벡터 데이터를 단일 시스템에서 관리 가능
    - SQL 표준으로 벡터 검색 접근 가능
  - 단점:
    - 순수 벡터 DB에 비해 벡터 검색의 성능과 확장성이 제한적
    - 대규모 벡터 컬렉션에서는 성능 저하가 발생할 수 있음

**Chroma DB:**

- 가장 많이 활용되는 오픈소스 벡터 DB
- 문서 저장 및 유사 문서 검색 기능 제공
- RAG 파일럿 테스트를 빠르게 실행할 수 있음
- Collection 기능: 텍스트 임베딩을 포함하는 상위 개념의 폴더로, 데이터 추가/교체/삭제와 같은 DB 관리 작업 수행 가능

#### 문서 검색기 Retriever

RAG에서 사용자의 질문과 답변 근거가 될 문서의 연결이 중요함. 이를 위해 다양한 모듈들이 있으며, 이를 Retriever라고 함.

**RAG에서 문서 검색이 중요한 이유:**

1. 사용자의 질문을 어떻게 해석할 것인가
   - 단순 유사도 검색만으로는 유연한 답변 생성이 어려움
   - 예: 생일 휴가를 물었을 때, 좀 더 포괄적으로 경조휴가 정보까지 제공하는 경우
2. 답변 근거가 될 문서를 어떻게/얼마나 가져올 것인가
   - 유사한 청크가 많다면 모두 가져와야 하는지 (컨텍스트 윈도우 문제)
   - 유사도 기준 N개만 추출하는 것보다, 다양한 근거를 포함해 다각도로 답변하는 것이 좋을 수 있음

**검색 방식 개선:**

**MMR(Maximum Marginal Relevance):**

- RAG 결과물로 더 풍부한 답변을 얻기 위해 단순 유사도 검색의 한계 극복
- MMR 공식은 두 부분으로 구성:
  1. 질문-문서 유사도: 사용자 질문과 주어진 청크의 유사도
  2. 문서-문서 유사도: 주어진 문서와 이미 선택된 문서 집합 중 가장 유사한 문서와의 유사도
- 최종 MMR 점수 = (질문-문서 유사도) - (문서-문서집합 내 가장 유사한 문서 유사도)
- 질문과의 유사도가 높으면서도 이미 선택된 문서와는 다른 문서가 높은 MMR 점수를 받음
- 람다(λ) 값으로 두 지표의 중요도 조절:
  - 람다 = 1: 질문과의 유사도가 가장 높은 문서 선택
  - 람다 = 0: 문서 집합 중 가장 유사한 문서와의 유사도가 낮은 문서 선택

**주요 Retriever 종류:**

- **MultiQueryRetriever**:

  - 사용자의 질문을 여러 버전으로 만들어 벡터 DB 내 검색을 원활하게 함
  - 질문 문장이 벡터 DB 내 청크들과 유사하지 않을 경우, 적절한 근거를 찾기 어려운 문제 해결
  - 사용자 질문을 여러 버전으로 변환해 검색 범위를 넓힘

- **MultiVectorRetriever**:

  - 문서의 벡터를 재가공하여 검색 품질을 향상시킴
  - 상위 문서 검색기(Parent Document Retriever)가 이 개념을 바탕으로 만들어짐
  - 청크 길이 문제 해결:
    - 청크가 너무 길면: LLM이 처리해야 하는 토큰 수가 늘어나 비용 증가 및 적절한 답변 찾기 어려움
    - 청크가 너무 짧으면: 맥락이 완전하지 않아 환각 현상 발생 가능
  - 상위 문서 검색기는 청크를 두 가지 기준으로 나눔:
    - 긴 길이의 상위 청크와 짧은 길이의 하위 청크
    - 검색 시 짧은 하위 청크로 정확한 결과 추출, LLM에게는 상위 청크를 전달해 완전한 맥락 제공

- **Long-Context Reorder**:
  - LLM의 컨텍스트 윈도우 크기가 늘어남에 따라 중요해진 기능
  - "Needle in a haystack" 평가지표 관련:
    - 컨텍스트 길이가 길어질수록 정보 추출 실패 확률이 높아짐
    - 그러나 맨 앞이나 맨 뒤에 위치한 정보는 100% 정확도로 추출 가능
  - 작동 원리:
    - 사용자 질문과 유사한 근거 문서를 여러 개 추출
    - 중요도 순으로 컨텍스트의 맨 앞쪽과 맨 뒤쪽에 배치
    - 이를 통해 답변의 정확도를 향상시킴
